# CPQ 技术架构设计

## 1. 推荐技术栈

如果由一个人完成前后端开发，建议使用以下技术栈：

| 层级 | 技术 | 选择原因 |
| --- | --- | --- |
| 前端 | React + TypeScript + Ant Design | 企业后台开发效率高，组件成熟 |
| 后端 | NestJS + TypeScript | 模块化清晰，适合中后台 API |
| ORM | Prisma | 数据模型清楚，迁移和类型支持较好 |
| 数据库 | PostgreSQL | 适合复杂关系数据，也支持 JSON 字段 |
| 缓存 | Redis，第一版可选 | 后续可用于缓存规则、会话、队列 |
| PDF 导出 | Puppeteer | 可以用 HTML 模板生成 PDF |
| Excel 导出 | ExcelJS | 适合生成报价明细 |
| 权限认证 | JWT + RBAC | 简单、通用、易扩展 |
| 部署 | Docker Compose | 适合单机或内网服务器部署 |

如果公司已有统一技术栈，也可以替换：

- Java 技术栈：Spring Boot + MyBatis/JPA。
- .NET 技术栈：ASP.NET Core + EF Core。
- Python 技术栈：FastAPI + SQLAlchemy。

## 2. 总体架构

```text
浏览器
  |
  | HTTPS
  v
前端应用
  |
  | REST API
  v
后端 API 服务
  |
  | ORM
  v
PostgreSQL 数据库

后端 API 服务
  -> PDF/Excel 导出服务
  -> 文件存储
  -> 操作日志
  -> 预留 CRM/ERP 集成接口
```

## 3. 后端模块划分

建议目录结构：

```text
src/
  auth/                登录、鉴权、权限守卫
  users/               用户管理
  roles/               角色与权限
  customers/           客户与联系人
  opportunities/       商机
  products/            产品、分类、SKU
  product-config/      产品配置和规则校验
  pricing/             价格本、折扣、税率、计价
  quotes/              报价和报价明细
  approvals/           审批流程
  exports/             PDF/Excel 导出
  orders/              订单
  audit/               操作日志
  common/              公共工具、异常、拦截器
```

### 3.1 登录与权限模块

职责：

- 用户登录
- Token 刷新
- 当前用户信息
- 修改密码
- 路由权限守卫
- 按钮权限判断

### 3.2 产品模块

职责：

- 产品分类管理
- 产品 CRUD
- 产品状态管理
- 产品选项定义
- 套餐产品定义

### 3.3 产品配置模块

职责：

- 校验用户选择的产品配置。
- 检查必选、互斥、依赖、数量、区域限制。
- 返回错误或警告信息。
- 计算选配项带来的价格调整。

第一版支持的规则可以先固定为：

```text
如果选择了 A，则必须选择 B
如果选择了 A，则不能选择 C
如果产品数量小于最小数量，则不允许保存
如果客户区域不在允许范围内，则不允许选择该产品
```

### 3.4 价格模块

职责：

- 价格本管理
- 价格明细管理
- 阶梯价格
- 折扣规则
- 税率规则
- 价格预览

报价计算必须以后端结果为准。前端可以做即时预览，但最终保存和审批必须使用后端计算结果。

### 3.5 报价模块

职责：

- 创建和编辑报价草稿。
- 添加报价明细。
- 调用配置校验。
- 调用价格计算。
- 锁定已审批和已发送版本。
- 复制报价或创建新版本。
- 管理报价状态流转。

### 3.6 审批模块

职责：

- 判断是否触发审批。
- 创建审批任务。
- 记录审批意见。
- 根据审批结果推动报价状态变化。

### 3.7 导出模块

职责：

- 生成报价 PDF。
- 生成报价 Excel。
- 保存导出文件。
- 记录导出日志。

## 4. 前端架构

建议目录结构：

```text
src/
  app/                 应用入口
  routes/              路由配置
  layouts/             主布局、登录布局
  pages/               页面
    dashboard/
    customers/
    opportunities/
    products/
    pricing/
    quotes/
    approvals/
    orders/
    settings/
  components/          公共组件
  services/            API 请求
  stores/              本地状态
  hooks/               自定义 Hook
  utils/               工具函数
  types/               TypeScript 类型
```

推荐库：

- React Router：页面路由。
- TanStack Query：接口请求、缓存、刷新。
- Zustand：复杂页面本地状态。
- Ant Design：表格、表单、抽屉、弹窗、步骤条。
- dayjs：日期处理。

## 5. 部署架构

第一版 Docker Compose 服务：

```text
cpq-web      前端静态站点
cpq-api      后端 API
cpq-db       PostgreSQL 数据库
cpq-storage  文件存储目录或简单文件服务
```

后续可增加：

```text
cpq-redis    缓存
cpq-worker   异步任务，如批量导入、复杂导出
```

## 6. 系统集成预留

### 6.1 CRM 集成

可能同步的数据：

- 客户
- 联系人
- 商机
- 报价状态

方向：

```text
CRM -> CPQ：客户、联系人、商机
CPQ -> CRM：报价金额、报价状态、成交结果
```

### 6.2 ERP 集成

可能同步的数据：

- 产品主数据
- 成本
- 库存
- 成交订单

方向：

```text
ERP -> CPQ：产品、成本、库存
CPQ -> ERP：客户接受后的订单
```

第一版可以只预留集成字段和接口，不做真实对接。

## 7. 核心架构原则

- 报价计算必须确定、可复现。
- 财务结果由后端统一生成。
- 已审批的报价使用快照保存。
- 规则校验结果必须给出人能看懂的原因。
- 状态变化使用明确动作接口，不允许随意改状态。
- 产品、价格、审批、报价四个模块边界要清楚。
