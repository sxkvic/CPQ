# CPQ 系统领导汇报版摘要

## 1. 建设目标

本项目建设一套企业内部 CPQ 报价系统，用于规范销售报价流程。

CPQ 代表：

- Configure：产品配置，选择型号、参数、模块、配件、服务。
- Price：价格计算，计算标准价、选配价、折扣、税费、成本、毛利。
- Quote：报价输出，生成正式报价单，并保留审批和版本记录。

系统建成后，销售可以在系统内完成从客户选择、产品配置、价格计算、折扣审批到报价单导出的完整流程。

## 2. 当前业务痛点

传统人工报价通常存在以下问题：

- 产品配置依赖销售个人经验，容易选错或漏选。
- 价格计算依赖 Excel，口径不统一。
- 折扣审批过程不透明，后期追溯困难。
- 报价单手工编辑，容易出现价格、税率、客户信息错误。
- 报价版本混乱，不清楚哪一版发给了客户。
- 成交后无法直接形成结构化订单数据。

CPQ 系统通过规则、价格、审批、文档模板的统一管理，解决这些问题。

## 3. 核心业务流程

```text
客户
  -> 商机
  -> 新建报价
  -> 选择产品
  -> 配置参数和选配
  -> 校验配置规则
  -> 自动计算价格
  -> 判断折扣和毛利是否需要审批
  -> 审批通过
  -> 导出报价单
  -> 客户接受
  -> 转订单
```

## 4. 系统主要模块

| 模块 | 说明 |
| --- | --- |
| 客户管理 | 管理客户、联系人、负责人、历史报价 |
| 商机管理 | 管理销售机会，并关联报价 |
| 产品管理 | 管理产品分类、SKU、产品属性、选项、套餐 |
| 配置规则 | 管理必选、互斥、依赖、数量、区域限制等规则 |
| 价格管理 | 管理价格本、阶梯价、折扣、税率、费用 |
| 报价管理 | 新建报价、配置产品、计算价格、版本管理 |
| 审批中心 | 处理超折扣、低毛利、大金额、手工改价审批 |
| 文档导出 | 导出 PDF 或 Excel 报价单 |
| 订单管理 | 将客户接受的报价转为订单 |
| 系统设置 | 管理用户、角色、权限、编号规则、报价模板 |

## 5. 第一版建议范围

第一版重点做“能真实使用”的闭环，不建议一开始就做过度复杂的平台。

建议第一版包括：

- 登录和权限控制
- 客户与联系人管理
- 产品和选项管理
- 价格本管理
- 报价编辑器
- 后端报价计算
- 折扣和毛利审批
- 报价 PDF 导出
- 报价版本快照
- 基础操作日志

第一版完成后，销售可以不用 Excel 手工算价，而是在系统内完成标准报价。

## 6. 推荐技术方案

如果由一个人完成前后端开发，建议采用以下技术栈：

| 层级 | 推荐方案 |
| --- | --- |
| 前端 | React + TypeScript + Ant Design |
| 后端 | NestJS + TypeScript |
| 数据库 | PostgreSQL |
| ORM | Prisma |
| 文档导出 | Puppeteer 生成 PDF，ExcelJS 生成 Excel |
| 权限认证 | JWT + RBAC |
| 部署 | Docker Compose |

这套技术栈开发效率高、结构清晰，比较适合一个人完成企业内部系统。

## 7. 交付周期预估

| 阶段 | 周期 | 目标 |
| --- | --- | --- |
| 第一阶段 | 6 到 8 周 | 完成可用 MVP，支持报价、计算、审批、导出 |
| 第二阶段 | 4 到 6 周 | 增强配置规则、套餐、导入导出、转订单 |
| 第三阶段 | 6 到 10 周 | 对接 CRM/ERP、多币种、审批设计器、报表分析 |

## 8. 成功标准

系统第一版上线后，应达到以下效果：

- 销售可以独立完成一份标准报价。
- 系统能自动检查基础产品配置错误。
- 折扣过高或毛利过低时自动触发审批。
- 审批记录完整可追溯。
- 报价单可直接导出为客户可见的 PDF。
- 已审批或已发送报价不能被直接修改。
- 修改报价必须产生新版本。

## 9. 实施优先级

建议按照以下顺序开发：

1. 用户登录和权限
2. 客户、联系人、产品等基础资料
3. 价格本和价格明细
4. 报价编辑器
5. 后端报价计算服务
6. 审批流程
7. 报价单导出
8. 版本快照和操作日志

## 10. 关键风险与控制

| 风险 | 控制方式 |
| --- | --- |
| 产品配置规则过早复杂化 | 第一版只支持固定类型规则 |
| 前后端价格结果不一致 | 后端计算结果作为唯一依据 |
| 审批规则不清晰 | 开发前先确定折扣、毛利、大金额阈值 |
| 报价模板频繁变化 | 报价模板单独管理，便于替换 |
| 业务要求一次性对接所有系统 | 第一版只预留接口，后续分阶段对接 |

## 11. 总结

CPQ 系统应以“报价”为核心建设。第一版不追求大而全，而是优先保障产品配置正确、价格计算准确、审批流程清晰、报价文档规范。

在第一版稳定后，再逐步扩展到 CRM、ERP、合同、订单、报表等更完整的商业中台能力。
