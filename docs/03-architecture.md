# CPQ Technical Architecture

## 1. Recommended Stack

For one full-stack developer, the recommended stack is:

| Layer | Choice | Reason |
| --- | --- | --- |
| Frontend | React + TypeScript + Ant Design | Fast enterprise UI delivery |
| Backend | NestJS + TypeScript | Structured modules, good validation, easy API development |
| ORM | Prisma | Clear schema, migration support, type-safe access |
| Database | PostgreSQL | Reliable relational model and JSON support |
| Cache | Redis, optional after MVP | Session, lock, and later rule cache |
| Export | Puppeteer for PDF, ExcelJS for Excel | Mature document generation |
| Auth | JWT + RBAC | Simple and extensible |
| Deployment | Docker Compose | Easy single-machine deployment |

Alternative backend choices:

- Spring Boot if the company uses Java.
- .NET if the company already has a Microsoft stack.
- FastAPI if the team prefers Python.

```text

  product-config/
  pricing/
  quotes/
  approvals/
  exports/
  orders/
  audit/
  common/
```

### 3.1 Auth Module

Responsibilities:

- Login
- Token refresh
- Current user profile
- Password update
- Route guards
- Permission decorators

### 3.2 Product Module

Responsibilities:

- Product categories
- Product CRUD
- Product status management
- Product option definitions
- Product bundle definitions

### 3.3 Product Configuration Module

Responsibilities:

- Validate selected options.
- Check dependency and exclusion rules.
- Return configuration errors and warnings.
- Calculate option price adjustments.

MVP rule model should support:

```text
IF selected_option contains A THEN require option B
IF selected_option contains A THEN exclude option C
IF quantity of product P < min_quantity THEN invalid
IF customer.region not in allowed_regions THEN invalid
```

### 3.4 Pricing Module

Responsibilities:

- Price books
- Price book items
- Tiered prices
- Discount rules
- Tax rules
- Pricing preview

Price calculation must run on the backend. The frontend can preview, but the backend result is the source of truth.

### 3.5 Quote Module

Responsibilities:

- Create and edit quote drafts.
- Add quote items.
- Run configuration validation.
- Run price calculation.
- Freeze approved and sent versions.
- Duplicate quote or create next version.
- Manage quote status.

### 3.6 Approval Module

Responsibilities:

- Evaluate approval triggers.
- Create approval tasks.
- Record approval decisions.
- Move quote status after approval result.

### 3.7 Export Module

Responsibilities:

- Render quote PDF.
- Render quote Excel.
- Store generated files.
- Record export logs.

## 6. Integration Design

### 6.1 CRM Integration

Possible sync data:

- Customers
- Contacts
- Opportunities
- Quote status

Integration direction:

```text
CRM -> CPQ: customer, contact, opportunity
CPQ -> CRM: quote amount, quote status, accepted result
```

### 6.2 ERP Integration

Possible sync data:

- Product master data
- Cost
- Inventory
- Accepted order

Integration direction:

```text
ERP -> CPQ: product, cost, inventory
CPQ -> ERP: accepted quote converted order
```

MVP can reserve integration tables and API endpoints without implementing real integration.

## 7. Core Design Principles

- Quote calculation must be deterministic.
- Financial results should be generated by backend services.
- Approved quote data should be snapshot-based.
- Rule evaluation should return human-readable reasons.
- API should expose clear status transitions instead of arbitrary status updates.
- Product, price, approval, and quote modules should stay separate.
